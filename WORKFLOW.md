# WORKFLOW: учебные спринты

Проект: сайт-каталог загородных домов из бруса на 1C-Bitrix «Стандарт» (демо),
Docker на macOS. Это учебная практика — работа идёт **спринтами с обучением**.

## Правила

1. **Каждый шаг объясняется ДО выполнения**: что делаем, где это лежит в
   Bitrix, зачем именно там, как это устроено под капотом — с аналогиями из
   жизни.
2. **После каждого спринта — полный СТОП**: отчёт «что сделали / что ты
   должен был понять / что дальше» + лог в Obsidian
   (`Claude Code/bitrix-practice/daily-changes/`), и ждём команды
   пользователя продолжать. Без команды следующий спринт не начинается.
3. Этот файл — рабочий регламент, сверяемся с ним каждый спринт.
4. Вопросы пользователя в любой момент важнее движения вперёд: остановиться,
   объяснить, показать в коде/админке.

Полная спецификация проекта (стек, инфоблоки, дизайн, анимации, ресурсы):
`~/.claude/plans/wiggly-enchanting-globe.md`, копия — в Obsidian
`Claude Code/bitrix-practice/context/`.

---

## Спринт 0. Окружение: поднимаем Bitrix в Docker

**Что изучаем**: из чего состоит Bitrix-хостинг (Nginx + PHP + MySQL и зачем
каждый), почему на Mac нужен Docker, структура файлов Bitrix.

**Аналогия**: Bitrix — многоквартирный дом от застройщика. `/bitrix/` — несущие
стены и инженерные коммуникации (трогать нельзя: «капремонт» — обновление —
всё затрёт), `/local/` — твоя квартира, где ты делаешь любой ремонт,
`/upload/` — кладовка с вещами жильцов (загруженные файлы). Docker — участок
с готовыми коммуникациями, на котором дом собирают за час.

**Что делаем**: клонируем BitrixDock; правки для Mac (убрать
`/etc/localtime:...:ro`; на Apple Silicon при проблемах с Percona —
`platform: linux/amd64` или MariaDB); `.env`: PHP 8.2, Nginx; кладём
`bitrixsetup.php` в webroot → мастер установки: демо «Стандарт», UTF-8, без
демо-контента. Разбираем каждый сервис compose и каждый шаг мастера.

**Готово когда**: главная и `/bitrix/admin/` открываются, «Проверка системы»
без критичных ошибок; пользователь понимает, где ядро, а где наш код.

**⏸ СТОП — отчёт, лог в Obsidian, ждём команды.**

## Спринт 1. Инфоблоки: данные о домах

**Что изучаем**: главная концепция Bitrix — инфоблоки; чем тип инфоблока
отличается от инфоблока, что такое разделы, элементы, свойства; ЧПУ; админка.

**Аналогия**: инфоблоки — архивная картотека. Тип инфоблока — шкаф
(«Каталог»), инфоблок — ящик в нём («Проекты домов»), разделы —
папки-разделители (Одноэтажные/Двухэтажные…), элементы — карточки (конкретный
дом), свойства — графы на карточке (цена, площадь…). Умный фильтр — поиск по
графам. ЧПУ — адресная табличка, по которой карточку находят с улицы.

**Что делаем**: создаём тип инфоблока и инфоблок `houses`, свойства из
спецификации (§4), разделы, SEO-шаблоны; скачиваем и жмём фото, рисуем
SVG-планировки, наполняем 6–8 домов. Показываем, как это выглядит в БД и в
админке.

**Готово когда**: все дома в админке с фото/планами/свойствами; пользователь
может сам создать элемент с нуля.

**⏸ СТОП.**

## Спринт 2. Шаблон сайта и дизайн-система

**Что изучаем**: как Bitrix собирает страницу (пролог → тело → эпилог), что
такое шаблон сайта, где живут CSS/JS/шрифты, дизайн-токены.

**Аналогия**: шаблон сайта — рамка с паспарту: `header.php` — всё, что выше и
левее фотографии, `footer.php` — всё, что ниже; сама страница (например,
`/about/index.php`) — фото, которое вставляют в рамку. Меняешь рамку — все
фото сразу выглядят по-новому.

**Что делаем**: шаблон `bruswood` в `local/templates/`, header/footer,
дизайн-токены (палитра/шрифты), меню (`bitrix:menu`), подвал, подключаем
локальные GSAP/Lenis/PhotoSwipe. Объясняем привязку шаблона к сайту.

**Готово когда**: каркас всех страниц в едином стиле, шрифты self-hosted;
пользователь понимает путь запроса от URL до footer.php.

**⏸ СТОП.**

## Спринт 3. Главная + анимации

**Что изучаем**: компоненты Bitrix и их шаблоны; как оживлять сайт GSAP-сценами.

**Аналогия**: компонент — стиральная машина: мотор и электроника (ядро
компонента в `/bitrix/components/`) не разбираем, а корпус и панель управления
(шаблон компонента) меняем — копируем шаблон к себе в
`local/templates/bruswood/components/` и перекрашиваем. Кеш компонента —
термос: суп не варят заново на каждый запрос.

**Что делаем**: секции главной, сцены на GSAP: SplitText-hero, parallax,
pin+scrub «этапы строительства», счётчики, stagger карточек. Каждую сцену
разбираем: триггер, таймлайн, зачем matchMedia.

**Готово когда**: сцены работают на 60fps, reduced-motion корректен;
пользователь понимает, из чего состоит вызов компонента и его шаблон.

**⏸ СТОП.**

## Спринт 4. Каталог с умным фильтром

**Что изучаем**: связка smart.filter + catalog.section, как фильтр строит
SEF-URL и достаёт данные из свойств, пагинация, сортировка.

**Аналогия**: фильтр — консультант в магазине: ты называешь бюджет и метраж,
он уходит к картотеке (инфоблоку) и приносит только подходящие карточки.

**Что делаем**: `/catalog/` с `catalog.smart.filter` + `catalog.section`,
кастомные шаблоны компонентов, карточки с иконками Lucide, мобильная
фильтр-панель.

**Готово когда**: фильтр по всем 5 свойствам возвращает корректные результаты
(сверяем с данными вручную).

**⏸ СТОП.**

## Спринт 5. Детальная страница дома

**Что изучаем**: catalog.element, работа с множественными файловыми свойствами
(галерея, планы), PhotoSwipe, хлебные крошки, SEO-шаблоны в действии.

**Что делаем**: детальная: hero, галерея + PhotoSwipe, SVG-планировки с
переключением этажей, таблица характеристик, комплектация, соседние проекты.

**Готово когда**: любой дом открывается со всеми блоками; лайтбокс работает с
жестами; title/description сгенерированы по шаблонам.

**⏸ СТОП.**

## Спринт 6. Контент, полировка, приёмка

**Что изучаем**: include-области (что сможет править контент-менеджер без
разработчика — сменные таблички на двери вместо ремонта), 404, View
Transitions, финальная верификация.

**Что делаем**: `/about/`, `/contacts/`, include-области (телефон, адрес),
404, favicon, View Transitions API, полный прогон чек-листа верификации.

**Готово когда**: весь чек-лист зелёный; итоговая документация в Obsidian.

**⏸ СТОП — проект завершён, финальный отчёт.**
